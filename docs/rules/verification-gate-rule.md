# Verification Gate Rule

구현→리뷰→수정 워크플로에서 **"무엇을 완료로 인정하는가"** 의 게이트 규칙. `docs/rules/subagent-rule.md`(오케스트레이션)와 `docs/rules/testing-rule.md`(테스트 작성)를 보완한다.

> 배경: Task 019에서 `reviewer`가 PASS를 냈지만 실제로는 **482개 기존 테스트가 빨간 상태**였다. 새 `@Service`가 새 리포지토리를 의존하게 되면서 컴포넌트 스캔으로 풀컨텍스트 `@SpringBootTest` 41개의 컨텍스트 로드가 깨졌는데, 구현 에이전트가 이를 "pre-existing 실패"로 잘못 단정했고, 정적 리뷰는 동적 회귀를 잡을 수 없어 게이트를 통과했다. 운(다음 단계의 우연한 발견)으로 잡힌 결함이다. 본 규칙은 그 공백을 메운다.

## 1. 리뷰 PASS ≠ 빌드 그린
- `reviewer`는 **정적 리뷰 전용**(코드 읽기)이다. 테스트를 실행하지 않으므로 **동적 회귀(컨텍스트 로드 실패·런타임 깨짐)를 탐지하지 못한다.**
- 따라서 **`reviewer`의 PASS를 단계 완료의 충분 조건으로 삼지 않는다.** PASS는 "plan 대비 코드가 옳다"는 정적 판정일 뿐이다.
- 단계 완료(done)는 **정적 게이트(reviewer PASS)** 와 **동적 게이트(전체 스위트 실제 실행 → 그린)** 를 **둘 다** 충족해야 한다.

## 2. 동적 게이트는 메인 에이전트가 직접 확인한다
- 각 구현 단계(backend/view) 및 수정(fixer) 후, 메인 에이전트는 **변경한 프로젝트에서 `./gradlew test` 전체를 실제로 실행해 `BUILD SUCCESSFUL`을 자기 눈으로 확인**한다(testing-rule §검증 실행과 동일 명령, 단 **확인 주체는 메인**).
- 서브에이전트가 보고한 "테스트 통과" 문장을 **그대로 신뢰해 게이트를 통과시키지 않는다.** 특히 서브에이전트가 자기 신규 테스트만 돌리고 전체 스위트를 돌리지 않았을 수 있다.
- 인프라(DB) 의존 통합 테스트가 섞여 전체 실행이 부담되면, 최소한 **회귀 신호(컨텍스트 로드 오류)** 를 잡는 표본을 돌리되, 단계 종료 전 1회는 전체 스위트 그린을 확인한다.
- **스키마 매핑 변경(Entity 신설·컬럼/타입 변경·마이그레이션 추가)이 포함된 단계에서는, 전체 스위트보다 먼저 `docs/rules/schema-mapping-validation-rule.md`의 전용 테스트를 단독 실행**해 JPA↔Flyway 정합부터 격리한다. 매핑 결함 하나가 무관한 통합 테스트를 대량으로 깨뜨려(entityManagerFactory 전체 실패) "환경 탓" 오진을 유발하기 때문이다.

## 3. "무관한/pre-existing 실패" 주장은 baseline 대조로 검증한다
- 서브에이전트(또는 누구든)가 "이 실패는 내 변경과 무관하다 / 원래 깨져 있었다"고 주장하면, **추정으로 받아들이지 말고 baseline과 대조해 증명한다.**
  - 방법: 변경을 `git stash -u`로 치우고(또는 변경 전 커밋에서) **동일 테스트를 실행**해 pre-change 상태에서 통과/실패를 확인한 뒤 복원한다.
  - pre-change에서 통과하던 테스트가 변경 후 실패하면 그것은 **회귀이며 이번 변경의 책임**이다.
- "실패 개수"가 크다고 자동으로 "환경/인프라 탓"으로 분류하지 않는다(019의 482건은 전부 회귀였다).

## 4. 가산적(additive) diff도 회귀를 만든다 — Spring 컴포넌트 스캔 주의
- "소스 diff가 삽입(additive)뿐이라 회귀가 없다"는 **잘못된 추론**이다.
- 새 `@Component`/`@Service`/`@Repository` 등 **빈을 하나 추가**하면, 운영과 동일한 컴포넌트 스캔으로 컨텍스트를 띄우는 **모든 풀컨텍스트 `@SpringBootTest`** 의 구성이 바뀐다.
- 새 빈이 **새 의존(예: 새 Repository)** 을 요구하는데, 리포지토리를 `@MockitoBean`으로 스텁(실 JPA 제외)하는 기존 테스트가 그 의존을 목하지 않으면 **컨텍스트 로드가 실패**한다.
- 따라서 **새 리포지토리/의존 빈을 추가하는 변경에서는, 그 의존을 스텁해야 하는 기존 풀컨텍스트 테스트 전부에 `@MockitoBean`을 함께 추가**한다(이 코드베이스의 확립된 관례 — 기존 테스트들이 `PaymentRepository`/`InventoryStockRepository`를 이미 그렇게 목하고 있다).
- 구현 에이전트는 자신이 만든 신규 테스트뿐 아니라 **이 파급(기존 테스트 갱신)** 까지 책임 범위로 본다. testing-rule(테스트 더블/컨텍스트 배선)과 함께 적용한다.

## 5. 게이트 미충족 시
- 동적 게이트(전체 그린)가 미충족이면, 그 단계는 **완료가 아니다.** 메인 에이전트는 원인을 규명(필요 시 §3 baseline 대조)하고 `fixer`로 회귀를 복구한 뒤 게이트를 재확인한다.
- `reviewer → fixer` 사이클 한도(subagent-rule, 최대 3회)는 정적 게이트 기준이며, **동적 게이트 미충족은 별도로 반드시 해소**한 뒤 사용자에게 보고한다.
- 게이트 통과/미통과, baseline 대조 결과, 회귀 복구 범위를 작업 보고에 남긴다(거짓 통과를 보고하지 않는다).
