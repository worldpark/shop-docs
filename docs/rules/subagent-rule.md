# Subagent Rule

## 서브에이전트 작업 규칙
- 메인 오케스트레이션(작업 분해, 서브에이전트 호출, 결과 취합, 최종 보고)은 반드시 메인 에이전트가 수행한다.
- 서브에이전트는 자신의 단일 책임만 수행하며 다른 서브에이전트를 직접 호출하지 않는다.
- 구현 단계는 `backend-implementor`(Spring Boot 백엔드)와 `view-implementor`(Thymeleaf 화면)로 분리한다.
- 부하/성능(k6) 작업은 `k6-implementor`가 담당한다(`shop-core/perf/k6/**` JS 스크립트). k6는 떠 있는 앱을 외부에서 가압하는 블랙박스 자산이라 `./gradlew test` 게이트 밖이며(머지 비차단), 애플리케이션 Java 코드·`build.gradle`을 수정하지 않는다. 검증은 메인의 풀 스위트가 아니라 **k6-implementor의 smoke 실제 실행(pass/fail)** 으로 한다(브라우저 E2E 실행 전담 `e2e-runner`와 구별 — k6-implementor는 스크립트 구현+smoke 검증).
- 메인 에이전트가 plan을 기준으로 작업을 배분하고, 구현 에이전트 간 인터페이스(모델 키, Service 시그니처, 폼 DTO, k6가 호출하는 REST 엔드포인트·요청 스키마)를 정합한다.
- 한 Task에 백엔드와 화면이 모두 필요하면 백엔드 -> 화면 순으로 호출한다. Service와 DTO가 먼저 정의되어야 화면이 안정적으로 바인딩된다.
- `reviewer -> fixer` 사이클은 한 Task당 최대 3회로 제한한다.
- 3회 안에 `reviewer`가 PASS를 내지 않으면 메인 에이전트가 사이클을 중단하고 남은 FAIL 항목을 사용자에게 보고한다.
- 사이클 반복 시 메인 에이전트는 매 회차 결과(PASS/FAIL, 잔여 항목, 수정 파일)를 기록한다.

## 서브에이전트 테스트 실행 범위 (풀 스위트 반복 금지)
> 배경/실증: Task 030에서 `fixer`가 통합 테스트 2개(파일 작성은 정상 완료)를 검증하며 **풀 스위트(`./gradlew test`, 1492개 + Testcontainers Postgres·Redis)를 반복 실행**해 1시간 넘게 묶였다. 결과물은 이미 디스크에 다 있었고, 느림의 원인은 순수 실행 비용(Docker 컨테이너 기동 × 풀 스위트 반복)이었다.

- 서브에이전트(`implementor`/`fixer`)는 작업 중 **자신이 만지거나 추가한 테스트만 타깃 실행**한다(`./gradlew test --tests "*Xxx*"`). 반복 피드백 루프에서 **풀 `./gradlew test`를 돌리지 않는다.**
  - 특히 Testcontainers(Docker) 통합 테스트가 섞이면 풀 스위트 1회가 수십 분~1시간이다. 이를 반복하면 비용이 폭증한다.
  - 테스트-전용 또는 국소 추가/수정일수록 타깃 실행으로 충분하다(프로덕션 코드 미변경 시 풀 회귀 위험은 낮다).
- **풀 스위트 그린 확인은 메인 에이전트의 최종 동적 게이트**다(`docs/rules/verification-gate-rule.md` §2). 서브에이전트는 이를 **중복 수행하지 않는다** — 단계 종료 시 메인이 1회 확인한다.
- 서브에이전트는 작업 보고에 **어떤 타깃 테스트를 돌려 그린을 봤는지**(명령·결과)를 남긴다. 메인은 그 보고를 신뢰해 게이트를 통과시키지 않고(verification-gate-rule §2), 풀 스위트를 직접 1회 실행해 확정한다.
- 메인 에이전트는 서브에이전트 호출 시 이 범위를 **명시**한다(예: "풀 스위트 금지, `--tests`로 해당 테스트만 실행").
