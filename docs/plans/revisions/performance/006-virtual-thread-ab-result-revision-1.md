# 가상스레드 A/B 측정 결과 (Task 006 Revision 1)

- 대상 Task: `docs/tasks/performance/006-performance-shop-core-virtual-thread-ab-evaluation.md`
- 측정 일자: 2026-06-16
- 측정자: 사용자(앱 토글 기동) + 메인(k6·서버측 지표)
- 목적: 가상스레드(VT) 도입/미도입을 동일 워크로드에서 A/B 측정해 **채택 근거**를 데이터로 확보.

## 측정 환경
- 워크로드: `catalog-read.js`(`GET /api/v1/products` + `/products/{id}`, 공개·읽기, 락 없음), **CONC_VUS=300 closed × 40s**, 깨끗한 카탈로그 시드.
- 토글: VT = 빈(`VirtualThreadConfig`, `--shop.threads.virtual.enabled=true`) / 풀 = `--spring.datasource.hikari.maximum-pool-size`. 셀마다 앱 재기동.
- 서버측: Micrometer/Prometheus(`jvm_threads_peak_threads`로 플랫폼 스레드 피크, `hikaricp_connections_max`로 풀 확인). VT 활성 교차검증 = 피크 스레드(가상스레드는 `ThreadMXBean` 미집계).
- **중요 한계**: k6·앱·PostgreSQL이 **동일 개발 머신에서 CPU 공유**. 절대수치(~900/s)는 비대표적(머신 전체 CPU 한계 추정). **VT vs 플랫폼 상대 비교는 동일 조건이라 유효**.

## 매트릭스 (풀10=단일런, 풀50=각 3런)
| 셀 | 풀 | 모델 | throughput | p95 | p99 | 피크 플랫폼 스레드 | read_5xx |
|---|---|---|---|---|---|---|---|
| 1 | 10 | 플랫폼 | 984/s | 442ms | 570ms | 277 | 0 |
| 2 | 10 | VT | 877/s | 472ms | 700ms | 91 | 0 |
| 3 | 50 | 플랫폼 | ~920/s | **599~634** (~620) | **830~905** (~865) | **281** | 0 |
| 4 | 50 | **VT** | ~915/s | **436~495** (~460) | **605~656** (~625) | **98** | 0 |

> 풀50은 각 3런: 플랫폼 p95={634,599,629}/p99={905,830,862}, VT p95={495,447,436}/p99={656,619,605}. **분포 비중첩**(플랫폼 최선 599/830 > VT 최악 495/656).

## 발견
1. **throughput은 풀·스레드 모델과 무관하게 ~900~980/s 평탄.** 풀 10→50도, 플랫폼→VT도 throughput을 못 올렸다. → 천장은 **CPU/처리능력**(동거 머신 CPU 공유 추정)이지 풀도 스레드 모델도 아니다. "풀 늘리면 VT 이김" 가설 기각.
2. **VT의 스레드 효율: 같은 300 동시를 플랫폼 ~280 스레드 vs VT ~90~98(약 1/3).** 풀 무관 일관.
3. **★ VT의 꼬리지연 이득(풀50, 각 3런 확정): p95 −26%(~620→~460), p99 −28%(~865→~625), 분포 비중첩 = 노이즈 아님.** 메커니즘: 300동시에서 플랫폼은 ~280 스레드를 만들어 (동거 머신) CPU 경합 → 컨텍스트 스위칭 오버헤드로 꼬리지연 악화. VT(98 스레드)는 경합이 적어 꼬리지연 개선 = 알려진 **"스레드 풀 포화 시 tail latency 개선"** 이득.
   - 단, **동거 머신이라 280-스레드 CPU 경합이 증폭**됨 → 전용 머신(코어 많음)에선 이득 폭이 작아질 수 있음(방향은 타당). 풀10은 단일런이라 비교 불충분.

## 결론 / 권장
- **throughput 이득은 없다**(CPU 바운드, ~920/s 동일). 단, **고동시성에서 (a) 꼬리지연 개선(p95/p99 ~26~28%↓) + (b) 플랫폼 스레드 ~1/3** 이라는 **실측 이득이 확인**됐다.
- 따라서 VT는 "성능(throughput) 도구"가 아니라 **"고동시성 자원 효율 + 꼬리지연 안정화 도구"**다. 채택 가치가 **무(無)는 아니다** — 동시성이 높아 플랫폼 스레드가 포화(수백)되는 운영 구간이라면 꼬리지연·자원 면에서 이득.
- **결정**: 현재 동시성·배포 규모에선 throughput이 핵심이고 그 이득이 없으므로 **미도입 유지가 여전히 합리적**(CLAUDE.md 정합). 단 **"VT는 무용"이 아니라 "현 시점 우선순위 아님"** 이 정확한 표현. 고동시성·tail latency가 운영 이슈가 되면 재평가(빈 기본 off로 보존돼 토글만 하면 됨).
- **throughput 이득이 생기는 별개 조건**: 풀 밖에서 오래 블로킹하는 경로(외부 서비스 호출 등) 추가 시. shop-core엔 아직 없음. **notification SMTP**(외부 소켓 블로킹)는 그 후보 → Task 007(단 SMTP 서버 한도가 진짜 천장, VT+rate바운드 필수).
