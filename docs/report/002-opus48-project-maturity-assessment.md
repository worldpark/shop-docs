## 종합 등급

**핵심 흐름은 준-프로덕션(near-production) 수준의 엔지니어링 규율을 갖춘 고급 포트폴리오/레퍼런스.**
단, **시스템 완성도(운영 가능성)는 중간**. 설계·테스트·문서 규율은 시니어급인데, 분산/운영 영역과 일부 품질 게이트에 사각이 있다.

한 줄 요약: 설계·테스트·문서 **규율은 시니어 포트폴리오 상위권**, 운영·분산·게이트 통합은 **미완성으로 명확히 남아 있는** 상태 — "잘 만들어지고 있으나 아직 운영 제품은 아닌" 단계.

## 강점 (근거)

- **아키텍처 규율**: Spring Modulith 모듈러 모놀리스 + `ModularityTests`로 경계 강제. **spi 의존 역전**(예: order→product.spi)으로 결합 통제. 단순 CRUD 더미가 아니라 경계가 실제로 검증된다.
- **이벤트 드리븐 정합성**: Transactional Outbox(Modulith Event Publication Registry) + Kafka 외부화, notification의 멱등 소비(`processed_event` 상태머신)·Resilience4j 서킷브레이커·DLQ. wire 직렬화 이중 인코딩 같은 함정까지 테스트로 방어(testing-rule에 실증 기록).
- **데이터 계층 엄격성**: Flyway가 스키마 소유 + `ddl-auto=validate`(ADR-007). Testcontainers 실 PostgreSQL 통합 테스트(H2로 못 잡는 동작까지). 동시성은 PESSIMISTIC_WRITE 직렬화.
- **테스트 피라미드 실재**: 단위(Mockito) + 슬라이스/통합(Testcontainers) + 브라우저 E2E(Playwright). 보안 매트릭스(401/403/404 IDOR 은닉)·동시성 RED→GREEN 경험 검증까지.
- **문서/결정 추적성**: ADR 9 · rules 14 · Task/plan 분리 워크플로우 · event-catalog SSOT. 결정의 "왜"가 남아 있어 포트폴리오로서 강하다.
- **보안·도메인 모델 성숙**: JWT + formLogin 이중 보안 체인, RoleHierarchy, IDOR(소유권 은닉) 일관, 도메인 팩토리/상태전이 메서드(Setter 금지), 개인정보 마스킹.

## 약점 / 리스크 (솔직)

- **품질 게이트의 사각**: 세션 중 실제 발견 —
  - `MultipleBagFetchException`(옵션 있는 주문 상세 조회 크래시 — 통합 커버 공백),
  - `reviews.rating` smallint↔int 매핑으로 ~150 테스트 연쇄 실패,
  - signup UI 변경이 **모든 signup 기반 E2E를 깨뜨렸으나 `./gradlew test` 게이트가 못 잡음**.
  → **E2E가 CI 게이트에 통합되어 있지 않음**이 구조적 약점.
- **운영/분산 미성숙(미완성 영역)**: 단일 브로커·단일 DB·인메모리 스케줄러, 분산락 미도입(ADR-005 보류), 관측성(메트릭/트레이싱/구조화 로그) 부재, 배포 토폴로지 미정(Eureka/Gateway/K8s 보류). "마이크로서비스 같은" 구조이지만 실제 운영 신뢰성 요소는 아직.
- **범위 과확장 신호**: `shop.domains.*.separated` 전송 토글(Task 034/035)은 데모로는 흥미로우나, **공유 DB·단일 배포물에서 REST 루프백**은 실익이 옅고 분산 트랜잭션 같은 본질 난제는 미해결. 학습 가치 vs 복잡도 비용의 균형 주의.
- **시각/계약 변경 파급**: 시스템 전역 KST 도입으로 **이벤트 wire가 `Z`→`+09:00`** 로 변경(ADR-009). Instant 컨슈머는 호환이나 **notification(별 레포) 미러링·계약 SSOT 동기화가 사람 손에 의존** — 크로스레포 계약 드리프트 위험.
- **외부 연동의 실체성**: PG는 mock, 결제·환불·재고 정합의 운영 경계(부분 배송 등)는 단순화. 데모엔 충분하나 "프로덕션"이라 부르긴 이르다.

## 하네스 엔지니어링(개발 방식) 자체 평가

- **강함**: planner → plan-reviewer → (사용자 승인) → implementor → reviewer → fixer → 게이트의 **다단계 검증 루프**가 실제로 결함을 차단한다. 예: Task 034 plan 단계에서 BLOCKER 3건 사전 차단, KST 회귀를 전체 게이트로 확인. rules/메모리로 재발 방지(smallint 매핑·`@Lazy` EntityManager 주입·Thymeleaf 예약어 등)를 시스템화.
- **약함/주의**: 검증이 **사람의 "진행/승인" 입력에 의존**하고, E2E·크로스레포 동기화가 자동 게이트 밖이라 사각이 남는다. 빠른 반복 중 signup UI 변경이 게이트를 우회한 사례가 그 증거.

## "완성"으로 가기 위한 우선순위

1. **E2E를 CI 게이트에 편입**(또는 최소 스모크 자동화) — 가장 큰 사각 제거.
   - **결정(2026-06-15): 보류.** 풀 E2E 게이트 편입은 하지 않는다. 이미 풀 `./gradlew test`가 ~19분인데 앱 기동 + 브라우저 E2E를 게이트에 얹으면 **task 작업 사이클이 과도하게 길어지고**(Playwright flaky 위험 가중), 1인 포트폴리오 단계에서 머지 차단형 게이트의 ROI가 낮다.
   - 대신 사각은 **가벼운 워크플로 규칙으로** 덮는다: Thymeleaf 템플릿/signup·로그인 등 화면을 건드린 task는 완료 전 해당 E2E를 `e2e-runner`로 수동 실행(게이트 시간 0). 필요 시 스모크 2~3개만 nightly(머지 비차단)로 한정한다.
2. **관측성·운영 기본**(메트릭/트레이싱/헬스/구조화 로그) + 배포 토폴로지 확정.
   - **결정(2026-06-15): 모든 기능 개발 완료 후 착수.** 기능이 계속 바뀌는 중에 계측 지점·배포 토폴로지를 먼저 잡으면 기능 변경마다 계측 자체를 재작업해야 한다. 기능 셋이 안정된 뒤 한 번에 붙이는 것이 재작업 비용이 낮다.
3. **크로스레포 이벤트 계약 동기화 자동 가드**(KST 같은 변경이 notification을 깨지 않는지 자동 검증).
   - **배경**: 계약이 세 곳에 분산 — 생산자(shop-core 발행 이벤트), SSOT(`docs/event-catalog.md`), 소비자(notification DTO, "event-catalog 필드 1:1 반영"을 **사람이 손으로 미러링**). 레포가 분리돼 어느 빌드도 셋을 대조하지 않아 드리프트가 런타임에야 드러난다. KST(ADR-009)는 wire 시각이 `Z`→`+09:00`로 바뀌었으나 소비자가 `Instant`라 운 좋게 호환됐을 뿐, 사람이 손으로 확인해 안 깨진 것.
   - **방향(결정, 2026-06-15)**: item 1·2와 같은 기조(무거운 자동화보다 가벼운 가드)로, 풀 Pact 인프라 대신 **골든 샘플 JSON 픽스처 공유**를 채택한다 — 대표 이벤트 JSON을 공유 위치에 두고 shop-core는 "발행 JSON이 픽스처와 일치", notification은 "그 픽스처를 역직렬화 성공"을 각자 테스트해 어긋나면 빌드 실패. 비용 대비 사각 축소가 가장 큼.
4. 결제/재고/배송의 **운영 엣지** 보강, 분산락(다중 노드) 결정.
5. 도메인 분리(034/035)는 **학습 데모로 명시**하거나, 추진 시 데이터 소유권/분산 트랜잭션까지 정직하게 다룰 것.
