# 프로젝트 평가 리포트 — shop (이커머스 모듈러 모놀리스 + 이벤트 드리븐 알림)

> 작성일: 2026-06-08
> 평가 대상: 상위 워크스페이스(`shop/`) 전체 — `shop-core`, `notification`, `docs/`
> 평가 방식: 정적 분석(소스 트리·빌드 설정·테스트·문서·git 히스토리) 기반. 실제 빌드/테스트 실행 검증은 미포함.
> 평가 맥락: 미완성 단계. **하네스 엔지니어링으로 개발된 결과물**의 수준을 정량·정성·프로세스 세 축으로 평가한다.

---

## 한 줄 총평

> **"포트폴리오"라는 목적 대비 과할 정도로 단단한 시니어급 백엔드.** 코드 품질·테스트 규율·아키텍처 일관성은 실무 프로덕션 기준에 근접하고, 특히 **개발 프로세스(하네스) 자체의 성숙도**가 코드보다 더 인상적이다. 다만 결제 도메인 미구현으로 간판인 "주문→결제→이벤트" 종단 흐름이 아직 닫히지 않았다.

## 2. 강점

### 아키텍처 일관성 (★★★★★)

- Spring Modulith 모듈러 모놀리스. `spi/`(포트)·`service/`(구현)·`Facade`·`ServiceResponse` 계층이 **모든 모듈에서 동일 패턴**으로 반복 — 일관성이 기계적으로 강제된다.
- `ModularityTests`·`*ModuleStructureTest`로 모듈 경계 위반을 테스트로 차단.
- Entity 절대 미노출, REST 컨트롤러 ↔ View 컨트롤러 분리, Kafka 단방향·Outbox 신뢰성 발행.
- 문서의 시스템 불변조건(`architecture.md` §9)이 코드에 실제로 반영됨.

### 코드 품질 — `OrderService`가 대표 증거 (★★★★★)

- variantId 오름차순 **비관적 락**(데드락 회피).
- 2단계 스냅샷 — advisory 사전검증 → 락 후 authoritative 재조회로 가격 권위 확정.
- 트랜잭션 **밖** bounded 재시도(orderNumber unique 충돌) + self-injection으로 `@Transactional` 프록시 적용.
- `SecureRandom` 기반 고엔트로피 주문번호, 404 존재 은닉(타인 주문 조회 방어).
- JavaDoc이 "무엇"이 아니라 **"왜"**를 설명(트랜잭션 분리 이유, self-injection이 필요한 이유). 신입~중급이 짜기 어려운 동시성 설계.

### 테스트 규율 (★★★★★)

- 테스트 LOC가 메인의 1.5배.
- Testcontainers 실 PostgreSQL 통합 테스트(예: PostgreSQL NULL 파라미터 타입 추론 검증).
- 컨트롤러별 보안 테스트, Thymeleaf 렌더링 테스트, **동시성 통합 테스트**(`OrderCreationConcurrencyIntegrationTest`).
- Playwright E2E를 별도 소스셋(`e2eTest`)으로 분리 — 인프라 비의존 `test` 원칙 보존.

## 3. 미완성 / 갭 (정직한 한계)

1. **결제 도메인 부재 → 간판 흐름이 닫히지 않음.** 아키텍처 문서가 내세우는 `OrderCompletedEvent`/`PaymentFailedEvent`/`ShippingStartedEvent`가 실제 도메인에서 발행되지 않고, `DummyEventController`(Outbox smoke test)로만 시연된다. **이벤트 드리븐 시연**이라는 목표가 아직 "배관만 검증, 실 트래픽 미연결" 상태.
2. **알림 실체 부재.** dispatch는 로깅뿐, dedup·발송이력 테이블은 백로그 009.
3. **"분산락"은 아직 표어.** 현재 동시성은 DB 비관적 락/atomic UPDATE로 처리(단일 인스턴스에선 정답). 백로그 007의 Redisson은 미착수 — 문서 표현이 구현보다 앞선다.
4. **메타 레포 커밋 위생.** 상위 레포 14커밋이 전부 `docs:` — 실제 코드는 중첩 `shop-core`(9커밋)·`notification`(1커밋)에 분리. 의도된 멀티레포 구조지만, 코드 진척이 상위 히스토리에 안 보여 외부에서 진척 파악이 어렵다.

## 4. 하네스 엔지니어링 관점 평가 — 이 프로젝트의 진짜 차별점

코드보다 **프로세스 산출물**이 더 성숙하다. "AI에게 코드 짜게 시킨 결과물"이 아니라 **재현 가능한 엔지니어링 파이프라인을 설계한 결과물**이다.

- **Task 생명주기 문서화**: `task → plan → revision`을 버전별로 보존. 설계 변경(예: 재담기 동시성을 atomic UPDATE로 정정한 revision)이 추적된다.
- **규칙을 SSOT로**: 13개 `*rule.md`(아키텍처·API 권한·에러 응답·이벤트 계약·금지 규칙…)가 에이전트 행동을 제약. CLAUDE.md가 "작업 전 해당 rule을 읽으라"고 강제.
- **`review-patterns.md` — 학습 루프의 증거**: 실제로 한 번 빠져나간 P1 결함(테스트 더블이 운영 배선을 가려 87개 테스트 통과했으나 앱 기동 실패)을 **재발 방지 체크리스트로 박제**하고, 회귀 테스트(`RefreshTokenStoreWiringTest`)까지 산출. 단순 자동화와 진짜 하네스 엔지니어링을 가르는 지점.
- **서브에이전트 오케스트레이션 규약**: `plan → backend/view-implementor → reviewer → fixer`(최대 3사이클) + `e2e-runner`, 책임 분리·중단 조건 명문화.
- **백로그 규율**: 이연 항목을 출처(어느 Task §몇)까지 달아 추적 → "몰라서 안 한 것"과 "알고 미룬 것"을 구분.

## 5. 종합 등급

| 축 | 등급 | 근거 |
|---|---|---|
| 아키텍처 설계 | **A** | Modulith 경계·포트/어댑터·Outbox·이벤트 계약 SSOT |
| 코드 품질 | **A** | 동시성·트랜잭션 설계가 시니어급, "왜" 중심 문서화 |
| 테스트 | **A+** | 테스트>메인 LOC, 다층(단위·실DB·보안·뷰·동시성·E2E) |
| **개발 프로세스(하네스)** | **A+** | task/plan/revision·rules·review-patterns 학습루프 |
| 기능 완성도 | **C+** | 결제 미구현으로 종단 이벤트 흐름 미완 |
| 운영 성숙도 | **B-** | 알림 로깅뿐, 분산락 미구현, 관측성 미확인 |

### 현재 수준 한 줄 결론

골격·규율·핵심 거래 흐름(상품→장바구니→주문)은 **프로덕션 근접 품질로 완성**됐고, 남은 건 "결제 도메인 구현 → 실 이벤트 발행 연결 → 알림 실 채널"이라는 **수직 슬라이스 1~2개**다. 이것만 닫으면 포트폴리오로서 "동작하는 이벤트 드리븐 시스템" 주장이 완전해진다.
